## 1. Cover Page

The cover page is the first impression. It should include:

- **Project Title** — Clear, descriptive name
- **Subtitle** — One-line description (e.g., "Product Requirements Document v1.0")
- **Goal Type** — ONE of: `Learning` | `Validating` | `Prototyping` | `MVP` | `Production` *(used by tech-project-forge to calibrate project complexity)*
- **Date** — Document creation/last update date
- **Author(s)** — Names and roles
- **Status** — Draft | In Review | Approved | Archived
- **Confidentiality** — Internal / Confidential / Public
- **Company/Team** — Organization name and logo reference

**Goal Type Guide:**
| Goal Type | When to Use |
|-----------|-------------|
| Learning | Exploring tech/domain, no user deployment planned |
| Validating | Testing a hypothesis with real users (concierge/wizard of oz) |
| Prototyping | Non-production demo to get stakeholder feedback |
| MVP | First shippable version with core features only |
| Production | Full-featured, scalable, production-grade system |

**Visual design:** The cover uses the primary color palette as accent. Large, bold typography for the project name. Clean layout with ample whitespace. DOCX cover generated by `generate_prd_docx.js`.

## 3. Executive Summary

**Purpose:** Give the reader everything they need to know in 2-3 paragraphs. A busy executive should be able to read only this section and understand what the project is, why it matters, and what's being asked of them.

**Structure:**
- Paragraph 1: What are we building and why? (the opportunity or problem)
- Paragraph 2: How will we approach it? (high-level solution and key differentiators)
- Paragraph 3: What do we need? (timeline, resources, key decision points)

**Example:**
> We propose building a real-time collaboration engine for our project management platform. Customer research shows that 73% of teams lose an average of 4.2 hours per week to context-switching between communication and task management tools. By embedding real-time co-editing, live cursors, and instant messaging directly into our task views, we can eliminate this friction and increase user engagement by an estimated 35%.
>
> The solution will leverage WebSocket connections for real-time state synchronization, with a conflict resolution layer based on CRDTs (Conflict-free Replicated Data Types). This approach scales horizontally and has been proven at companies like Figma and Notion. We will deliver in three phases over 16 weeks.
>
> We are requesting approval for a team of 4 engineers, 1 designer, and 1 PM for Q3. Key decision needed by [date]: choice between building on existing WebSocket infrastructure vs. adopting a managed service (Ably/Pusher).

## 4. Problem Statement

**Purpose:** Clearly articulate the problem you're solving. Ground it in evidence — user research, support tickets, churn data, competitive pressure.

**Framework:**
- **Who** is affected? (specific user segments)
- **What** is the problem? (observable behavior or gap)
- **When/Where** does it occur? (context and frequency)
- **Why** does it matter? (business impact, quantified if possible)
- **Current workarounds** — how do people cope today?

**Tips:**
- Use data: "42% of users abandon the onboarding flow at step 3"
- Quote users: "I spend 20 minutes every morning just figuring out what changed overnight" — Enterprise Customer
- Show the cost of inaction: "At current churn rate, we lose $2.4M ARR annually from this issue"

## 5. North Star Metric

**Purpose:** The single most important metric that defines whether the product is succeeding. Aligns the entire team on what matters most. Every feature decision should be evaluated against its impact on the North Star.

**Format:**
```
North Star Metric: [metric name]
Definition:        [what exactly is measured]
Current Baseline:  [current value]
Target:            [goal value]
Measurement Date:  [by when]
Tracking Tool:     [how it's measured]
```

**Example:**
```
North Star Metric: Weekly Active Habits Completed
Definition:        Number of habit check-ins completed per week by active users (users who logged in at least once in last 7 days)
Current Baseline:  N/A (new product)
Target:            10,000 completions/week by end of Month 3
Measurement Date:  90 days post-launch
Tracking Tool:     Amplitude "habits.checkin" event, weekly cohort view
```

**Why one metric?** Multiple competing north stars create political alignment problems. When every team has a different "most important thing," roadmap prioritization becomes a negotiation instead of a decision. One north star forces clarity.

**Supporting Metrics (Guardrails):** Add 2-3 guardrail metrics that should NOT decline while optimizing the north star (e.g., "retention must not drop below 40% D7 while pursuing engagement growth").

## 6. Goals & Success Metrics

**Purpose:** Define what success looks like, measurably. Every goal should have a metric, a current baseline, a target, and a timeframe.

**Format:**

| Goal | Metric | Current | Target | Timeframe |
|------|--------|---------|--------|-----------|
| Reduce onboarding drop-off | Completion rate | 58% | 80% | 90 days post-launch |
| Improve user engagement | DAU/MAU ratio | 0.23 | 0.35 | 6 months |
| Decrease support burden | Tickets per user/month | 1.8 | 0.9 | Q4 |

**Anti-patterns to avoid:**
- "Improve user experience" — not measurable
- "Make it faster" — no target or baseline
- "Increase revenue" — too vague, no timeframe

## 7. Scope Definition

**Purpose:** Draw a clear boundary. What's in, what's out, and what's deferred.

### In Scope (v1.0)
List specific features and capabilities included in this release. Be explicit.

### Out of Scope
List things that are explicitly NOT included. This prevents scope creep.

### Future Considerations (v2.0+)
Ideas that are valuable but deferred. This shows you've thought about them without committing.

**Why "Out of Scope" matters:** Without it, every stakeholder assumes their pet feature is included. Disagreements surface during development instead of during planning. Save yourself the pain.

## 9. User Stories + Job Stories (JTBD)

**Purpose:** Translate the solution into specific, testable requirements using both traditional User Stories (what the user wants) and Job Stories / JTBD (why they want it).

**Traceability IDs:** All user stories use ID format `US-XXX`. Goals/metrics use `GM-XXX`. These IDs are referenced in ERD and Milestone docs for traceability.

### Format

**User Story format:** "As a [persona], I want [capability] so that [benefit]"
**Job Story / JTBD format:** "When [situation], I want to [motivation], so I can [expected outcome]"

Use Job Stories for features where the WHY matters more than the WHO — they're more outcome-focused and harder to game.

Each requirement should include:
- **ID** — Unique identifier (`US-XXX`)
- **User Story** — Traditional format
- **Job Story** — JTBD format (for key features)
- **Acceptance Criteria** — BDD Given/When/Then format
- **Priority** — P0 (Must Have), P1 (Should Have), P2 (Could Have), P3 (Won't Have this version)
- **Effort Estimate** — T-shirt size (XS, S, M, L, XL) or story points

### Example

| ID | User Story | Job Story (JTBD) | Priority | Effort |
|----|-----------|-----------------|----------|--------|
| US-001 | As a team lead, I want to see real-time cursor positions of collaborators so that I know who's working on what | When I'm reviewing a task, I want to know if a collaborator is already editing it, so I can avoid conflicting with their work | P0 | L |
| US-002 | As a member, I want to receive instant notifications when someone edits my task so that I stay informed | When I'm away from a task I own, I want to be notified of changes, so I can respond or course-correct without checking manually | P1 | M |
| US-003 | As an admin, I want to set permissions for who can edit vs. view so that sensitive tasks are protected | When managing sensitive project data, I want granular access control, so I can share progress without risking unauthorized changes | P0 | L |

### Acceptance Criteria Example (US-001) — BDD Format

```gherkin
Given a user is viewing a shared task board
When another collaborator moves their cursor on the same board
Then the cursor position updates within 100ms with the collaborator's name and avatar

Given 50 users are simultaneously active on a board
When all users move their cursors
Then all cursor positions render without frame drops below 30fps

Given a collaborator has been inactive for 30 seconds
When the idle timer expires
Then their cursor fades out with a 500ms opacity transition
```

**Why BDD?** Given/When/Then acceptance criteria are directly translatable to automated test cases (Cucumber, Behave, SpecFlow). This eliminates ambiguity between product intent and QA verification, reducing the "it works on my machine" gap.

## 10. User Flows

**Purpose:** Step-by-step narrative of primary user journeys through the product. Required for tech-project-forge compatibility — the forge skill reads these to understand navigation and feature interactions. Complements user stories (which describe what users want) by describing how users actually move through the system.

**Format:** Numbered steps with decision points. Include the "happy path" and 1-2 error/edge paths per flow.

**How many flows?** One flow per major use case. For an MVP, typically 3-5 flows.

### Flow 1: New User Registration and Onboarding

**Actor:** New visitor (unauthenticated)
**Entry point:** Landing page or direct URL

**Happy Path:**
1. User opens app at `/` (landing page)
2. User clicks "Get Started" or "Sign Up" button
3. App navigates to `/sign-up`
4. User enters email + password (or clicks social OAuth button)
5. Auth provider validates credentials and creates account
6. App redirects user to `/onboarding`
7. User completes onboarding form (name, preferences, etc.)
8. App creates user profile in database via webhook
9. App redirects to `/dashboard` — user sees empty state with "Create your first [item]" prompt
10. User creates first item → dashboard populates

**Error Path: Email already registered**
- At step 4: auth provider returns "email already in use" error
- App displays: "An account with this email exists. [Sign in instead?]"
- User is offered sign-in link

**Error Path: Network failure during account creation**
- At step 8: webhook delivery fails
- App shows loading state, retries 3x with backoff
- After 3 failures: show error "Something went wrong. Please try refreshing."
- Support: user can re-trigger profile creation from `/settings`

## 11. Error States & Edge Cases

**Purpose:** Every major feature must document what happens when things go wrong. Missing error state documentation is the #1 cause of poor user experience — developers make up error handling on the fly and it's inconsistent.

**For each major feature, document:**
1. Network failure / timeout
2. Empty state (no data, first-time user)
3. Invalid input
4. Permission denied / unauthorized
5. Rate limit reached
6. Conflict (e.g., concurrent edits, duplicate submission)

### Error State Template

**Feature:** [Feature name]

| Error Scenario | User Message | System Action | Recovery Path |
|---------------|-------------|--------------|---------------|
| Network timeout on submit | "Save failed. Check your connection and try again." | Queue for retry, show retry button | Auto-retry after 5s; user can retry manually |
| Empty state (no items) | "No [items] yet. Create your first one!" | Show empty state illustration + CTA | "Create [item]" button |
| Validation: required field missing | "Please enter a [field name]" | Highlight field in red, focus it | User corrects field |
| Unauthorized access | "You don't have permission to view this." | 403 response, redirect to dashboard | "Go to dashboard" link |
| Rate limit reached | "You've reached the limit for [action]. Try again in [time]." | 429 response with Retry-After header | Show countdown timer |
| Duplicate submission | "You've already [action] this [item]." | 409 response | Explain state, offer alternative action |

### Global Error Handling Standards

- All error messages written in plain language — no error codes to end users
- All errors offer a recovery action (retry, contact support, or navigate away)
- Technical error details logged server-side, never shown to users
- Form validation: inline errors next to the field, not top-of-form banner
- Network errors: show retry button with exponential backoff (5s, 10s, 30s)

## 10. Telemetry & Analytics Hooks

**Purpose:** Ensure observability is planned from day one — not bolted on after launch. Every major feature should be instrumented before code review.

### Event Catalog

| Event Name | Trigger | Properties | Priority |
|-----------|---------|------------|----------|
| `feature.used` | User activates a key feature | `feature_id`, `user_role`, `session_id` | P0 |
| `onboarding.step_completed` | User completes an onboarding step | `step_number`, `duration_ms` | P0 |
| `error.encountered` | Unhandled exception or user-facing error | `error_code`, `page`, `stack_hash` | P0 |
| `export.requested` | User exports data | `format`, `row_count` | P1 |

### Key Funnels

Define the critical user journeys to track as funnels:
- **Onboarding funnel:** Sign-up → Profile setup → First action → Return visit
- **Conversion funnel:** Free trial → Feature X used → Upgrade prompt → Payment
- **Engagement funnel:** Login → Dashboard view → Core action → Session end

### Dashboard Requirements

Specify what dashboards are needed for launch monitoring:
- Real-time active users and error rates
- Feature adoption rates (day 1, day 7, day 30)
- Funnel drop-off visualization
- Performance metrics (p50, p95, p99 latency)

### Instrumentation Plan

| Aspect | Decision |
|--------|----------|
| SDK / Library | Segment, Amplitude, Mixpanel, or custom |
| Event Schema | Noun.verb format (e.g., `report.generated`) |
| Data Destinations | Analytics warehouse, real-time dashboard, alerting |
| PII Handling | What is excluded, anonymized, or hashed |
| Sampling | Full capture vs. sampling rate for high-volume events |

## 14. Privacy & Compliance

**Purpose:** Define privacy practices and regulatory requirements from day one. Retrofitting GDPR compliance after launch costs 10x more than building it in. This section feeds directly into the ERD's LINDDUN threat model and security section.

**Applicable Regulations (check all that apply):**
- [ ] GDPR (EU/UK — any EU users or processing EU data)
- [ ] CCPA (California — B2C apps with California users)
- [ ] HIPAA (US — health data)
- [ ] PCI DSS (payment card data — handled by Stripe if using Stripe)
- [ ] SOC2 (enterprise SaaS — required for many B2B sales)
- [ ] COPPA (US — users under 13)

**PII Inventory:**
| Data Element | Collection Purpose | Storage Location | Retention Period | Right-to-Delete |
|-------------|-------------------|-----------------|-----------------|----------------|
| Email address | Account identity, login | `users.email` | Account lifetime | Yes — cascade delete |
| Display name | Profile, collaboration | `users.display_name` | Account lifetime | Yes — anonymize |
| Payment info | Billing | Stripe (not our DB) | Per Stripe policy | Via Stripe |
| Usage data | Analytics, product improvement | Analytics tool | 2 years rolling | Aggregated — no PII |

**User Rights Implementation:**
- **Right to Access** — User can export all their data: `GET /api/users/me/export` returns JSON
- **Right to Delete** — `DELETE /api/users/me` hard-deletes all PII within 30 days
- **Right to Rectification** — `PATCH /api/users/me` allows updating any PII field
- **Consent** — Cookie consent banner for analytics; email opt-in for marketing (double opt-in)
- **Data Portability** — Export as JSON or CSV on request

**Data Handling Practices:**
- No PII in logs or error messages
- No PII passed to AI models (anonymize before sending to Claude/OpenAI)
- DPIA (Data Protection Impact Assessment) required before adding new PII collection
- DPA (Data Processing Agreement) signed with all third-party processors (Clerk, Stripe, Resend)

## 15. Accessibility Requirements

**Purpose:** Accessibility is not optional for production software. WCAG 2.1 Level AA is the legal minimum in most jurisdictions and should be the default standard.

**Minimum Standard:** WCAG 2.1 Level AA

**Requirements:**

| Category | Requirement | Success Criterion |
|----------|------------|------------------|
| Keyboard Navigation | All interactive elements must be reachable and operable with keyboard only | WCAG 2.1.1, 2.1.2 |
| Focus Indicators | Visible focus indicator on all interactive elements (min 3:1 contrast ratio) | WCAG 2.4.7 |
| Color Contrast | Text: min 4.5:1 ratio; Large text (18pt+): min 3:1 ratio | WCAG 1.4.3 |
| Alt Text | All meaningful images have descriptive alt text; decorative images have `alt=""` | WCAG 1.1.1 |
| Form Labels | All form inputs have associated `<label>` elements | WCAG 1.3.1 |
| Error Identification | Form errors identified in text (not color alone) | WCAG 1.4.1, 3.3.1 |
| Screen Readers | ARIA labels on all interactive elements without visible text labels | WCAG 4.1.2 |
| Motion | Animations respect `prefers-reduced-motion` media query | WCAG 2.3.3 |
| Text Resize | Content usable at 200% text size without horizontal scrolling | WCAG 1.4.4 |
| Language | `lang` attribute set on `<html>` element | WCAG 3.1.1 |

**Testing:**
- Automated: `axe-core` in Playwright E2E tests (catches ~30% of WCAG issues)
- Manual: Keyboard-only navigation testing before each release
- Screen reader: Test with VoiceOver (macOS) and NVDA (Windows) on key flows
- Lighthouse: Accessibility score >= 90 on all pages (run in CI)

|------|-----------|--------|----------|------------|

**Severity Guide:**

- **High** — Significant scope reduction or timeline slip

- **Low** — Minor inconvenience

## 18. Launch Checklist

**Purpose:** Pre-launch verification that the product is ready across all dimensions. Engineering, product, marketing, legal, and support must all be ready for a successful launch.

### Product Readiness
- [ ] All P0 and P1 features in scope are shipped and working
- [ ] No open P0 or P1 bugs
- [ ] All acceptance criteria verified on staging
- [ ] Performance benchmarks met (p95 < 500ms, LCP < 2.5s)
- [ ] Error rate < 0.1% on staging under load test

### Analytics & Monitoring
- [ ] All telemetry events firing and appearing in analytics dashboard
- [ ] North Star Metric is being tracked and baseline established
- [ ] Alerting rules configured (P0/P1 paging thresholds set)
- [ ] Dashboards set up for launch day monitoring
- [ ] Runbooks written for top 5 likely incident scenarios

### Legal & Privacy
- [ ] Privacy policy written and linked from app
- [ ] Terms of service written and linked from app
- [ ] Cookie consent banner implemented (if applicable)
- [ ] GDPR/CCPA compliance review completed
- [ ] DPAs signed with all third-party data processors

### Marketing & Support
- [ ] Landing page live with accurate feature descriptions
- [ ] Help documentation written for core user flows
- [ ] Support team trained on product features
- [ ] Launch announcement drafted (email, social, blog)
- [ ] Feedback collection mechanism in place

### Technical Operations
- [ ] Production environment provisioned and tested
- [ ] CI/CD pipeline deploying to production successfully
- [ ] Database backups configured and tested (restore tested)
- [ ] Feature flags for all new features tested ON and OFF
- [ ] Rollback procedure documented and tested on staging
- [ ] On-call rotation established with escalation path
